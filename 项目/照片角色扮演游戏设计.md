# 照片角色扮演游戏设计

> 基于 llama.cpp + Qwen3.5 量化版，Windows 本地运行
> 一张照片 → 识别角色 → 注入人设 → 自动对话 → 推进剧情

## 硬件环境

- **Windows 10**，llama.cpp webui 已跑通
- **显卡：** P106-100 6G
- **模型：** Qwen3.5 量化版（多模态，可识图）

## 核心架构：三个角色

```
调度器（Python 脚本，调同一个 llama.cpp API）
    │
    ├── 角色A（演员）
    ├── 角色B（演员）
    └── 叙事者（上帝视角，掌握情节线）
```

### 轮流调用方案（方案 C）

**不是多实例（显存不够），不是单 prompt 双角色（角色不隔离），而是单实例分步推理：**

1. 每次只调一次 llama.cpp，只扮演一个角色
2. 轮流喂不同的 prompt（每个角色只看到自己的私有设定 + 公开对话历史）
3. 角色完全隔离——不知道对方的内心想法

### 调度流程

每轮循环：
1. 检测是否"卡住"（冷场检测）
2. 如果卡住 → 叙事者出手，扔一个微小扰动
3. 角色A回合 → 调 llama.cpp → 输出台词+动作
4. 角色B回合 → 调 llama.cpp → 输出台词+动作
5. 写入公开历史，回到 1

### 角色输出格式

角色每轮输出包括两种内容：
- **动作/表情**（角色自己描述自己）
- **台词**（对话内容）

格式示例：
```
（低头看手表，犹豫了一下）好吧，我告诉你。
```

**规则：** 角色只能描述自己的动作，不能说对方在做什么。

### 公开历史管理

- 保留最近 6-8 轮对话
- 早期剧情定期摘要压缩（节省 token）
- 格式统一，叙事事件用 `[叙事]` 标记

## 角色隔离策略

| 方案 | 优点 | 缺点 | 结论 |
|------|------|------|------|
| A. 单 prompt 双角色 | 省资源，连贯 | 角色隔离差，会串戏 | ❌ |
| B. 多实例独立进程 | 隔离完美 | 显存翻倍，P106 扛不住 | ❌ |
| **C. 单实例分步推理** | **隔离好，省资源** | **节奏慢一倍** | **✅ 推荐** |

## 叙事者的角色定位

### 之前：冷场救火员
- 对话卡住时扔一个事件
- 描述环境气氛
- 每轮必出场 → 话太多，喧宾夺主

### 最终方案：后台待命，卡住才出手
- **平时不出场**，让角色自由发挥
- **只在卡住时介入**，扔一个微小扰动
- 卡住检测用**规则判断**（字数短、重复率高、连续无进展），不额外调 LLM
- 设"介入冷却"——出手后至少等 3 轮才能再出手

### 上级升级：掌握情节线（半静态方案）

叙事者掌握整条情节线，但不是靠 LLM 动态生成，而是：

**你手动写死的剧情节点 + 调度器规则检测 + 叙事者只负责写事件文字**

```
静态剧情线（你写）：
├── 阶段1: 试探（2-4轮）→ 关键词"画"触发
├── 阶段2: 冲突（3-5轮）→ 关键词"承认"触发
└── 阶段3: 真相（自由发挥）

检测逻辑：调度器用规则判断关键词，不靠 LLM
事件描述：LLM 动态生成（保留鲜活感）
```

| 维度 | 纯动态 | 半静态（推荐） |
|------|--------|----------------|
| 剧情线 | LLM 自己编 | 你写死 |
| 节点触发 | LLM 判断 | 规则检测 |
| 事件描述 | LLM 写 | LLM 写 |
| 节奏控制 | 不可控 | 你说了算 |
| 稳定性 | 差 | 好 |
